# flowerss-bot

Telegram RSS bot. Fork of rssflow/flowerss-bot. Go + telebot v3 + GORM (SQLite default, MySQL optional).

## Quick commands

```bash
make build              # build binary
make test               # go test ./... -v
go test ./... -v        # same thing
go test ./internal/bot/handler/ -run TestRemove -v  # targeted test
```

## Architecture

```
main.go → core.NewCoreFormConfig() → bot.NewBot(core) → scheduler.NewRssTask(core)
                                          ↓                        ↓
                                   Bot.registerCommands()    task.Register(bot)  # bot implements RssUpdateObserver
                                   Bot.Run() → tb.Start()   task.Start()        # polling loop in goroutine
```

**Runtime loop:** Scheduler polls RSS sources every `config.UpdateInterval` minutes. New content → `bot.SourceUpdate()` → `BroadcastNews()` sends messages with inline unsubscribe buttons to each subscriber.

**Key type:** `session.Attachment` (protobuf) encodes `{user_id, source_id}` into inline button callback data. Hex-encoded, prefixed by telebot as `\f<unique>|<hex>`. ProcessUpdate strips the prefix before the handler sees it.

## Domain model

- **Source** — an RSS feed URL. Has `ErrorCount`; paused when `ErrorCount >= config.ErrorThreshold`.
- **Subscribe** — links a `UserID` (Telegram chat ID) to a `SourceID`. `UserID` is the chat where `/sub` was issued (user's private chat ID or group chat ID).
- **Content** — individual feed items. Deduped by `HashID` (derived from source link + GUID + item link).

## Critical invariants

- `core.Unsubscribe()` deletes the subscription first, then does best-effort source cleanup. Source cleanup errors are logged, not propagated — the subscription delete is what matters to callers.
- `SubscriptionExist` + `DeleteSubscription` use `(user_id, source_id)` as the compound key. No soft deletes — `EditTime` has only `CreatedAt`/`UpdatedAt`, no `DeletedAt`.
- Inline button callback data is `session.Attachment` protobuf → hex. The `Unique` field routes telebot callbacks; `Data` field carries the hex payload. Max callback_data is 64 bytes (Telegram limit); largest case (negative group chat ID) is ~49 bytes.

## Gotchas

- **Middleware chain is global:** `UserFilter → PreLoadMentionChat → LoadUserLanguage → IsChatAdmin` runs on ALL handlers including button callbacks. `IsChatAdmin` auto-passes for private chats.
- **Button handlers don't call `ctx.Respond()`** — Telegram shows a spinner that times out after ~30s. Not a bug, but noticeable UX quirk.
- **Success message uses HTML** (`unsub_success_button_format`): `source.Link` and `source.Title` are interpolated raw. Feed titles with `<` or `>` could break the HTML parse and cause the edit to fail silently.
- **`ctx.Edit()` without ReplyMarkup preserves existing inline buttons** — after unsubscribing, the button stays visible on the edited message.
- **i18n:** Translations loaded from `/opt/flowerss/locales` (Docker) or `./locales` (local). Missing keys return `[translation missing for key: ...]` strings. Test translations load from `../../locales` relative to handler test files.

## Testing

Mock storage interfaces live in `internal/storage/mock/` (generated by mockgen from `storage.go`). Handler tests use hand-written mocks in the test file itself and `editCaptureCtx` to intercept `ctx.Edit()`.

To regenerate storage mocks: `go generate ./internal/storage/`

## Deployment

Docker image built from `Dockerfile` (golang:1.25-alpine builder → alpine runtime). Locales copied to `/opt/flowerss/locales`. Config at `/root/.flowerss/config.yml`. Data at `/root/.flowerss/data.db` (SQLite).

Runs on a Tailscale host. SSH access requires correct key — check `~/.ssh/config` for host aliases.
